How it works

Five stages. Same order, every run.

This is a real sequence, not a diagram for decoration — each stage reads the previous stage's output file and can't run out of order.

01

Extract

Pulls raw transaction history — either from Etherscan's live API or a synthetic generator for testing — and writes it untouched. Nothing is cleaned here on purpose: the raw file is always available to debug against.

src/extract.py → data/raw/transactions.json
02

Transform

Validates every row against a required schema, drops what's incomplete, deduplicates on transaction hash, and converts wei → ETH and unix time → readable timestamps. This is also where the integrity percentage is actually computed.

src/transform.py → data/processed/transactions_clean.csv
03

Load

Writes clean rows into an indexed SQLite table — indexes on address, block, and time mean filtered queries don't scan the whole table. Also builds the daily summary view the dashboard reads from.

src/load.py → data/processed/onchain.db
04

Quality

Scores the run across four independent dimensions — see the integrity methodology page for the exact formula — instead of collapsing everything into one vague percentage.

src/quality.py → data/processed/quality_report.json
05

Analytics

Flags statistical outliers by z-score, computes a 7-day rolling trend, and fits a least-squares regression to forecast the next 3 days of volume. Simple methods, chosen deliberately — explainable beats clever for a baseline.

src/analytics.py → data/processed/analytics.json
Underneath

One scheduler, no manual steps

An APScheduler job runs stage 1 through 5 once on boot, then again on a fixed interval, forever. Every run writes a status file — that's what the dashboard's "last run / next run" line is reading. Nobody has to remember to click anything.

On-chain ETL and analytics, woven clean.
